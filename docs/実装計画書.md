## 1. 企画の技術的な読み替え

実装の優先順位を決めるために、企画のどこが本体かを先に確定させておく。

**このゲームの本体は演出である。**

ほじりシーンのメカニクスを分解すると「穴に指を入れる → 2.5秒待つ → 離す → 抽選結果が出る」しかない。ゲージ速度はグリグリしてもしなくても同じ、と企画で明言されているため、プレイヤーの入力が結果に影響する余地は意図的にゼロにしてある。つまり**ほじりの面白さは、抽選の引きと、ほじっている最中の手応えだけで支えられている**。

> **【後日変更】ゲージは「動かした量で溜まる」方式に変えた**（NIKKE の指揮官ぬいぐるみ方式）。§5.1 に詳細。
> 上の分析は「入力が結果に影響しない」という前提から演出への全振りを導いたものだが、**その前提自体を作者が覆した**。動かさないと1ミリも進まないので、グリグリは演出ではなく操作になった。
> ただし**演出が本体である**という結論は変わらない。溜め方が変わっただけで、出てくるものは相変わらず抽選任せだからである。

したがって「Phase 5: 演出」は仕上げの磨きではなく、**それ自体が機能**である。工数配分もそのつもりで組む（§9）。

同じ理由で、以下は実装時に死守する:

- ~~**グリグリは損得ゼロだが、やった方が気持ちいい**を担保する。~~ → 動かした量で溜まる方式にしたため、この心配は消えた（やらないと進まない）。ただし**グリグリ量に応じて、鼻の穴の歪み・画面揺れ・鼻くそが飛び出す瞬間の勢い・SEの太さを派手にする**のはそのまま残す。「進むから動かす」だけでなく「動かすと気持ちいい」を両方成立させる。
- **デカい鼻くそを引いた喜びを、後から否定しない**（§5.2 で詳述）。

## 2. 技術選定

| 項目 | 採用 | 理由 |
|---|---|---|
| 言語 | Vanilla JS (ES Modules) | ビルド不要。push したらそのまま Pages で動く |
| 描画 | Canvas 2D + DOM のハイブリッド | アクション系は Canvas、UI系は DOM |
| 音 | Web Audio API で手続き生成 | 音声ファイル不要。「ポン！」は合成で作れる |
| 絵 | 手続き生成 + Canvas パス | 画像アセット不要（§4.1） |
| 保存 | LocalStorage | サーバ不要 |
| 入力 | Pointer Events | マウス／タッチを 1 系統で処理できる |
| フレームワーク | なし | 提出物が「html など」なので、読める形で完結させたい |

**ビルドツールを入れない**のが方針の中心。npm も webpack も挟まないので、リポジトリの中身がそのまま提出物になり、GitHub Pages の設定も「main ブランチの root を配信」だけで済む。

### ハイブリッドの切り分け

- **Canvas**: ほじり、飛ばし（鼻の変形、カメラ追従、パーティクル、画面揺れ）
- **DOM**: コレクション、ショップ、タイトル、リザルト（スクロール・レイアウト・テキストが主役）

コレクションの各セルだけは小さな `<canvas>`（96×96）を敷いて、鼻くその形状を1回だけ描く。

## 3. 全体アーキテクチャ

### 3.1 座標系とレイアウト

**論理解像度 720 × 1280（9:16）に固定**し、CSS `transform: scale()` で画面にフィットさせる。

```
scale = min(innerWidth / 720, innerHeight / 1280)
```

すべてのチューニング数値（当たり判定、エッジ帯の幅、鼻の位置）を論理座標で書けるので、端末差を考えなくてよくなる。PC では左右に黒帯が出るが、スマホ縦画面想定なので想定通り。

### 3.2 シーンの十字配置

3×3 のワールド（2160 × 3840）を用意し、十字の5セルだけ使う。

```
                (1,0) 飛ばし
  (0,1) ショップ  (1,1) ほじり  (2,1) コレクション
                (1,2) リザルト／タイトルへ
```

`#world` に `transform: translate(-cellX*720, -cellY*1280)` をかけるだけで遷移できる。CSS transform なので GPU 合成に乗り、スライドは滑らかになる。

**この配置には副次的な利点がある。** 縦スクロールするシーン（コレクション・ショップ）の復帰エッジが左右方向になるため、スクロール操作とエッジ判定の軸が直交し、競合しない。配置は仮置きとのことだったが、この2つは左右のままにしておくことを推奨する。

### 3.3 エッジ遷移

企画の未確定点その1（触れた瞬間か／閾値か）への回答は **閾値方式 + プレビュー**。

| パラメータ | 初期値 |
|---|---|
| エッジ帯の幅 | 62px（画面幅の約8.6%） |
| プレビュー最大量 | 86px（約12%） |
| 確定閾値 | 帯の奥62%（画面端から24px以内）に到達、または帯内に280ms滞在 |
| スライド時間 | 300ms ease-out |

帯に入ると隣のシーンが最大12%ぬるっと覗き、指を戻せば元に戻る。この「覗き」が**何があるかを教えるチュートリアルを兼ねる**ので、説明文を置かずに済む。

**四方にエッジタブを常時出し、それ自体を押しても飛べるようにした。** 押し込み量はタブのバーが満ちる形で返すので、「バーが満タン＝遷移が確定」が目で分かる。

タブを足したのは、押し込み方式が**行き先を知っている人には遅い**ため。毎回ふちまで運んで閾値を越えるのを待たされる。押せるようにすれば、覚えた人は1タップで飛べて、覚えていない人は今までどおり覗きながら進める。両方を同じUIで賄える。

タブの実装で気をつけた点:

- **`pointer-events` は `.show` が付いている間だけ `auto`。** アクション中はタブごと消えるので、叩いても反応しない（`locked` の判定と二重に守る）。
- **`stopPropagation()` する。** `#stage` まで伝えると、シーン側が「画面を押した」と受け取り、飛ばしシーンの角度決めなどを巻き込んで発火する。
- 遷移は `go()` を通すので `armed` が落ち、**滞在判定と二重に発火しない**。

`locked` フラグを持ち、ほじり中・角度決め中・ゲージ溜め中・飛行中は `true` にしてエッジ判定を丸ごとスキップする（企画の要求どおり）。

**再武装（`armed`）が要る。** 確定したら、いったん帯から出るまで次の遷移を受け付けない。実装時に2つの実害が出た:

- ショップから右エッジで鼻に戻ると、指が右エッジに残っているせいで**着いた瞬間に図鑑まで飛ばされる**（ピンポン遷移）。
- ロック中も `armed` を落とさないと、グリグリで指が端まで流れたまま**ほじりが終わった瞬間に隣へ飛ばされる**。企画が要求した「操作中の誤遷移を防ぐ」が、アクション明けに結局破られてしまう。

**下（リザルト）だけは確定後にもう一段の確認を挟む**。誤爆したときの被害が唯一大きいセルなので、ここだけ非対称にする。

**DOMシーンのボタンは、左右パディングをエッジ帯より広く取る**（72px > 62px）。帯に食い込むと、ボタンを押した指がそのまま遷移を誘発する。

### 3.4 ループ

`main.js` に `requestAnimationFrame` を1本だけ持ち、アクティブなシーンの `update(dt)` / `render()` を呼ぶ。遷移中は移動元・移動先の2シーンを更新する（飛ばしシーンの指のメトロノームが、覗いた瞬間に止まっていると死んで見えるため）。

`dt` は 1/30 秒でクランプする。タブを離れて戻ったときに物理が吹っ飛ぶのを防ぐ。

### 3.5 入力

Pointer Events に一本化する。

- `pointerdown` で `setPointerCapture` → 画面外に出ても離した瞬間を取り逃さない
- `#stage` に `touch-action: none` → スマホのスクロール／ピンチが割り込まない
- 画面座標 → 論理座標の変換は `input.js` に閉じ込め、各シーンは論理座標だけ見る

## 4. データ設計

### 4.1 鼻くそマスタ（`src/data/boogers.js`）

```js
{
  id: 'b012',
  name: 'かちこち鼻くそ',
  size: '大',              // 小 / 中 / 大 / 特大 / 伝説
  rarity: 'SR',            // N / R / SR / UR / SECRET
  desc: '乾燥しきって岩のようになった一品。ぶつけると痛い。',
  seed: 48271,             // ← 形状の種
  secret: false,
}
```

**形状は seed から手続き生成する**（`core/shape.js`）。`mulberry32(seed)` で乱数を作り、円周24点の半径を複数のサイン波の重ね合わせでゆらして、ぼこぼこした閉曲線を作る。

この方式を採る理由は、企画の「**まだほじっていない鼻くそも、シルエットは図鑑に表示される**」という要求を素直に満たせるからである。形状がマスタの `seed` から決まるので、未所持でも同じ形が計算でき、シルエットと実物が必ず一致する。画像を32枚描いて管理する必要もない。

構成（総数39。図鑑のカウントは通常枠の37のみで、シークレットは別枠）:

| レア度 | 通常枠 | シークレット | 出現率（強化なし） |
|---|---|---|---|
| N | 9 | - | 59.5% |
| R | 7 | - | 24.8% |
| SR | 10 | - | 11.0% |
| UR | 7 | - | 3.5% |
| **GOD（四神）** | **4** | - | **0.8%** |
| SECRET | - | 2 | 0.5% |

**GOD は四神（青龍・朱雀・白虎・玄武）専用の階層**で、UR の上に置いた。シークレットと違い**図鑑に枠が見える**のが肝で、「あと1体」が見えているからこそ揃えたくなる。隠したらセットとして機能しない。

4体で枠を分け合うぶん1体あたりはさらに薄くなるので、`rarityBias` を 7 と高くして**ほじり力の強化がはっきり効く**ようにした。これで「四神を揃えたいなら鼻ほじり力を上げろ」という導線ができ、それまで抽象的だった `pick` 強化に具体的な目的が生まれる。

四神コンプに必要なほじり回数（実測・60回試行の平均）:

| ほじり力 | 必要回数 |
|---|---|
| 強化なし | 1,062 |
| +50% | 350 |
| +150% | 207 |

4体とも **伝説サイズで統一**してある。引くだけでなく「飛ばせるようにする」（＝パワー強化400回）まで含めて終盤の目標になる。

#### 四神コンプの祝い（`godfest`）

4体目を引いた瞬間、通常のリビールに続けて専用の演出に入る。**四神は東西南北を司るので、画面の四方にそのまま配置する**（上=玄武/北、右=青龍/東、下=朱雀/南、左=白虎/西）。集めた4体が意味のある並びに収まるので、ただ4つ並べるより効く。

- 中心から0.45秒おきに1体ずつ、所定の方角へ飛んで着席
- 4体そろった瞬間にフラッシュ＋ヒットストップ＋ファンファーレ、`四神 制覇` を金グラデで表示
- 一時金 30,000コイン（強化が一段飛ぶ額）
- 6秒で自動終了。3.2秒以降はタップで飛ばせる（長い演出を毎回見せられるのは苦痛なので）

発火は `gain()` が返す `godComplete` フラグで、**「直前が3体で、今4体になった」瞬間だけ true**。`seen` は永久なので二度と発火しない（2回目以降は通常のコインだけ、を実測で確認済み）。

サイズはレア度と**独立した軸**にする（企画で「レア度のほかに、大きさのステータスがある」と分けられているため）。「よく出るくせにバカでかい鼻くそ」や「激レアなのに極小の鼻くそ」が存在する方が図鑑として面白い。

### 4.2 セーブデータ（LocalStorage: `hojirinus.save.v1`）

```js
{
  version: 1,
  seen:   ['b001', 'b007', ...],        // 図鑑：見た記録。永久。消えない
  stock:  { b001: 12, b007: 3, ... },   // 在庫：実体。飛ばすと減る
  coins:  1240,
  upgrades: { pick: 12, power: 30, speed: 8 },   // 購入回数
  records: { 小: 41.2, 中: 55.0, 大: 0, 特大: 0, 伝説: 0 },  // サイズ別自己ベスト
  stats:  { totalPicks: 210, totalFlies: 44 },
}
```

**`seen` と `stock` を別テーブルにしているのが、この設計の要**。企画の「図鑑への登録は永久に残り、実体はそれとは別に在庫として個数を持つ」がそのまま構造になっている。飛ばしても `stock` だけが減り、`seen` は触らない。

保存は変更時に 500ms デバウンスして書く。毎フレーム書くと重い。
`version` を持たせておき、将来マスタを増やしたときにマイグレーションできるようにする（在庫が消えると心が折れるので、ここは最初から入れておく）。

## 5. ゲームロジック

### 5.1 ほじり

**動かした量でゲージが溜まる**（NIKKE の指揮官ぬいぐるみ方式）。当初の「グリグリしてもしなくても 2.5秒」から変更した。

```js
mv     = min(|ポインタ移動量|, digCapPerSec * dt)   // フレーム落ちでの荒稼ぎを潰す
moving = mv >= digMoveMin                          // これ未満は「止まっている」
gauge += mv / (digDist / (1 + upRate('dig')))      // ← 動かした量
       + (moving ? digAssist * (1 + upRate('dig')) * dt : 0)   // ← 動いている間の最低保証
```

`digAssist`（最低保証）が要る理由は実測で出た。移動量だけで組むと、**ちまちま動かす人と振り回す人で7倍以上の差**がつき、控えめな人は20秒経っても溜まらなかった。動いている間だけ加算されるので、**止めればゼロ**という原則は保たれる。

実測（`digDist=3600` / `digAssist=0.12`）:

| グリグリの激しさ | 満タンまで | ほじりスピード+100% |
|---|---|---|
| 静止 | **永久に溜まらない** | 同じ |
| ちまちま | 6.1s | 3.1s |
| ふつう | 4.1s | — |
| 激しく | 2.4s | — |

激しく動かしたときが約2.4秒で、当初の 2.5秒とほぼ一致する。テンポを落とさずに、操作を意味のあるものにできた。

- 開始条件: 指先が鼻の穴のヒット領域内にある状態で `pointerdown`
- ほじり中は「**ほじれ！**」を出す。手を止めるほど文字が大きく跳ね、色が赤に寄り、`指を動かさないと溜まらない` が出る。動かさないと進まない仕様は、言わないと伝わらない
- **触れるまでは指を鼻より手前に描く。** 常に穴の暗がりへ沈めると、狙っている最中に指先が消えてどこを指しているか分からなくなる。沈めるのはほじり始めてから
- 開始後は指を穴に固定し、ポインタとの差分ぶん**鼻がゴムのように伸びて歪む**。指は抜けない
- 深さ（ポインタY）は開始条件と演出にのみ使う。抽選には影響させない
- **満タン前に離す → スカ**。鼻くそは出ず、指に何もついていない絵と気の抜けたSEだけ出す（0.6s）。無反応より、スカの方が情報量があって面白い
- 満タン後に離す → 抽選 → 出現

#### 抽選

レア度 → エントリの2段抽選。ほじり力 `p`（= 購入回数 × 0.005）が両段に効く。

```js
// 1段目：レア度
baseW      = { N: 60, R: 25, SR: 11, UR: 3.5, SECRET: 0.5 }
rarityBias = { N: 0,  R: 1,  SR: 2.5, UR: 5,  SECRET: 8 }
w = baseW[r] * (1 + p * rarityBias[r])

// 2段目：そのレア度内のエントリ
sizeBias = { 小: 0, 中: 0.5, 大: 1.5, 特大: 3, 伝説: 5 }
w = entryWeight * (1 + p * sizeBias[size])
```

N の重みだけ据え置き、上位の重みを増やす。正規化の結果として N が相対的に減る。**1つのステータスで「レア度アップ」と「デカさアップ」の両方を賄える**ので、企画のステータス3種構成をそのまま実装できる。

`p = 1.0`（200回購入）で UR の重みは 3.5 → 21、約6倍になる。

### 5.2 飛ばし

**角度フェーズ**: 指を横から見た図で 0°〜90° を往復（周期2.2s）。クリック／タップで確定。

> 周期は 1.6s から 2.2s に下げた。メトロノームは正弦で振れるので**45°が最速点**になり、1.6s だと 40〜45° の帯を通過するのに 28ms（1.7フレーム）しかなく、運ゲーだった。
> ただし `sin(2θ)` は45°付近が平坦（40°でも98.5%）なので、**距離のためには精度が要らず、ヒットストップの気持ちよさのためだけに精度が要る**という構造になっている。狙う価値はあるが、外しても損はしない。
**ゲージフェーズ**: 長押しで 0→1 を 1.2s で充填。満タン後も押し続けられる（1.0でクリップ）。押している間は画面が振動し続け、**満タンかつ 40〜45° のときだけヒットストップ**が入る。離した瞬間に発射。

#### 飛距離

```js
sizeBonus = { 小: 1.0, 中: 1.3, 大: 1.7, 特大: 2.2, 伝説: 3.0 }
reqPower  = { 小: 20,  中: 28,  大: 38,  特大: 48,  伝説: 60 }

playerPower = 20 * (1 + upgrades.power * 0.005)
powerRatio  = clamp(playerPower / reqPower[size], 0, 1)
effBonus    = 1 + (sizeBonus[size] - 1) * powerRatio   // ★
v0 = 18 * (1 + upgrades.speed * 0.005) * gauge * effBonus
d  = v0 * v0 * sin(2 * θ) / 9.8
```

★の行が企画の「序盤はパワーの上限が低いため、デカい鼻くそは本来の飛距離を出せず、ステータス強化によって初めて活きる」の実装。パワーが足りないほど `sizeBonus` の旨みが 1.0 に潰れていく。

> **`reqPower` の幅は `basePower` の3倍以内に収めること。**
> 強化は +0.5% の**加算**なので、現実的な購入回数（〜400回）で `playerPower` はせいぜい3倍にしかならない。ここを開きすぎると伝説の「パワー不足」が永久に解けず、ショップの「やがて赤字が解ける」という約束が嘘になる。
> （初稿では `reqPower[伝説] = 150` としていたが、これだと強化を1300回買わないと充足率1.0に届かず到達不能だった。実装時に実測して発覚し、上の値に直した。）

実測（`power` と `speed` を同数だけ買った場合）:

| power購入 | 充足率 小/中/大/特大/伝説 | 小 | 伝説 | 累計コスト |
|---|---|---|---|---|
| 0回 | 100 / 71 / 53 / 42 / 33 | 33m | 92m | 0 |
| 100回 | 100 / **100** / 79 / 63 / 50 | 74m | 298m | 7,604 |
| 200回 | 100 / 100 / **100** / 83 / 67 | 132m | 720m | 32,788 |
| 300回 | 100 / 100 / 100 / **100** / 83 | 207m | 1,469m | 115,918 |
| 400回 | 100 / 100 / 100 / 100 / **100** | 298m | 2,678m | 390,086 |

サイズが1つずつ順番に解禁される梯子になっている。ショップの赤字が5回に分けて解ける。

**パワー不足でも、小より下回らせない**（`effBonus` の下限は 1.0）。ここは意図的な判断で、下回らせると「デカいの引いたのに損」になり、企画の中心にある『デカい鼻くそをひり出す快感』と正面から衝突する。デカい鼻くそは常に嬉しく、ただし**本気を出せていない**、という状態にする。

また、**45°がベスト**という企画の指定は `sin(2θ)` の性質から自動的に満たされる。40° は `sin(80°) = 0.985` なので約98.5%。ヒットストップの窓を 40〜45° にする指定とも、そのまま整合する。

#### 飛行

カメラは鼻くそを追従。距離が伸びたら徐々にズームアウト（下限あり）。
**着地したら跳ねずにベチャッと止まる**。鼻くそは粘着質だから、というのが理由になっていて、物理も単純化できる。バウンド／転がりは拡張候補として残す。

### 5.3 コイン

| 契機 | 額 |
|---|---|
| ほじり | N:2 / R:6 / SR:20 / UR:70 / SECRET:200 |
| 図鑑・初登場ボーナス | N:30 / R:80 / SR:250 / UR:800 / SECRET:2000 |
| 飛ばし | `floor(d * 2.5)` |
| 自己ベスト更新 | 飛ばし分を ×2 |

**ほじりの収入は伸びず、飛ばしの収入は強化とともに伸びる**という形にしてある。ほじり収入は約 50〜150コイン/分で頭打ちだが、飛ばしは序盤 83コイン/回 → 伝説解禁後 6,700コイン/回 まで伸びる。

これで序盤はほじりが生活費を稼ぎ、強化が進むと飛ばしが主収入になり、「強化 → 飛ぶ → 稼ぐ → 強化」のループが自走する。初登場ボーナスを厚くしてあるのは、図鑑コンプという緩いゴールに経済的な意味を持たせるため。

コイン単価を 0.8 → 2.5 に上げてある。ゴールも競争相手も無いゲームなので渋くする理由がない。面白いのはほじりと飛ばしであって、貯金ではない。

### 5.4 強化

```js
cost(n) = floor(baseCost * 1.012 ** n)
baseCost = { pick: 50, dig: 45, power: 40, speed: 60 }
// 1回で +0.5%
```

企画の3種に **ほじりスピード（`dig`）** を足して4種になった。ゲージが移動量制になったことで「グリグリが面倒」という新しい不満が生まれ、それを金で解決する先が要るため。ショップの `KEYS` は `Object.keys(CFG.ups)` から作る（手書きのリストに追加し忘れて、買えないステータスが生まれるのを防ぐ）。

`dig` だけは％表示では効きが分からないので、**必要グリグリ量の実数**（3600 → 2988 …）も併記する。減っていく数字が見えれば「指が楽になった」と伝わる。

**セーブの移行に注意。** `load()` は `Object.assign(S, fresh(), d)` で `S.ups` ごと古い値に置き換えるため、新キーが欠けたまま残り、`buy()` の `S.ups[k]++` が `NaN` になって**そのステータスが二度と買えなくなる**。`CFG.ups` の全キーを走査して埋め直す処理を入れてある。

伸び率は 1.045 → 1.012 →（後述）**1.008** と2回下げている。

#### なぜコイン倍率では直らなかったか

「お金稼ぎが遅い」に対して最初に浮かぶのは**コイン効率アップのステータス追加**だが、実測するとそれでは直らないと分かった。100回強化するごとの倍率を並べるとこうなる。

| 強化回数 | コストの伸び | 収入の伸び |
|---|---|---|
| 〜100 | 3.30倍 | 3.24倍 |
| 〜200 | 3.30倍 | 2.42倍 |
| 〜300 | 3.30倍 | 2.04倍 |
| 〜400 | 3.30倍 | 1.82倍 |
| 〜500 | 3.30倍 | **1.36倍** |

**コストは指数関数（常に3.30倍）、収入は多項式（伸びが減衰）。** 飛距離は `v²` なので伸びるように見えるが、`effBonus` が `sizeBonus` の上限で頭打ちになると、あとは `(1+r)²` でしか伸びない。指数 vs 多項式なので必ず発散する。実際ちょうど100回付近で逆転し、100回強化するのに必要な投数が **163 → 166 → 226 → 364 → 659** と膨らんでいた。

コイン効率アップは**線形の倍率**なので、この発散を止められない。曲線を右にずらすだけで壁の位置が変わるだけである。加えて、自己言及的（コインを稼ぐためにコインを払う）なので**必ず最初に全振りするのが正解**になり、他4つの選択が意味を失う。

そこで `upGrowth` そのものを寝かせた。

| upGrowth | 0→100 | 100→200 | 200→300 | 300→400 | 400→500 |
|---|---|---|---|---|---|
| 1.012（旧） | 163 | 166 | 226 | 364 | **659** |
| **1.008（採用）** | 129 | 89 | 82 | 89 | **108** |
| 1.006 | 116 | 65 | 49 | 44 | 44 |

1.006 以下は終盤が楽になりすぎて強化のありがたみが消えるので、**一定に保てる最小の変更**として 1.008 を採った。

#### 図鑑ボーナス

あわせて、**図鑑の完成度がそのままコイン倍率になる**ようにした（最大2.0倍）。

```js
dexMul = 1 + (通常枠の収集率 × 0.6) + (四神コンプ ? 0.2 : 0) + (シークレット数 × 0.1)
```

図鑑は初回ボーナスを取ったら経済的な役目が終わっていて、39種＋四神まで広げたのに勿体なかった。集める行為がそのまま稼ぎに効くようにして、「ほじる→集める」と「飛ばす→強化」の両輪を噛み合わせる。ショップの枠を増やさずに済むのも利点。

倍率は**図鑑画面とショップの両方に常時表示**する。見えない倍率は集める理由にならない。

上2つを入れた後の必要投数（図鑑の進みを強化と連動と仮定）:

| 強化 | 図鑑 | 倍率 | 1投 | 必要投数 |
|---|---|---|---|---|
| 0回 | 9/37 | ×1.15 | 263 | 113 |
| 100回 | 16/37 | ×1.26 | 936 | 71 |
| 200回 | 22/37 | ×1.36 | 2,442 | 60 |
| 300回 | 29/37 | ×1.84 | 6,741 | 49 |
| 400回 | 35/37 | ×1.90 | 12,720 | 57 |

序盤が一番きつく、進むほど楽になる形になった。

## 6. 演出（＝本体）

### ほじり

- 指の深さとグリグリ加減で**鼻の穴が歪む**（穴のパスの制御点をポインタ方向に押す）
- 勢いよくほじると画面が揺れる（グリグリの角速度に比例）
- 出る瞬間、スッポンが外れるような「ポン！」（Web Audio: サイン波のピッチ急降下 + ノイズバースト + ローパス）
- **初登場ならアイリスアウト → 後光 → レア度ぶんの星** が出る
- グリグリ量に応じて、上記すべての強度が上がる

### 飛ばし

- ゲージ満タン + 40〜45° でヒットストップ + 画面振動が一段強くなる
- 発射時にヒットストップ 80ms
- 着地の「ベチャッ」

### 共通（`core/juice.js`）

画面揺れ、ヒットストップ、アイリスアウト、ヒットストップ中の時間停止（`dt = 0` を配るだけ）をここに集約する。各シーンから `juice.shake(power, time)` の形で呼ぶ。

スマホでは Vibration API を併用し、ゲージ溜め中に実際に端末を震わせる。

## 7. 未確定点への提案

### (1) エッジ発火方式 → §3.3 の閾値 + プレビュー方式

### (2) 強化後の即時フィードバック

**+0.5% は、単体では絶対に体感できない。** そこを直接どうにかしようとせず、4方向から埋める。

1. **長押し連打購入**。押しっぱなしで連続購入され、SEのピッチが1段ずつ上がっていく。ほじりと飛ばしが両方「長押しで溜める快感」なので、**ショップにも同じ気持ちよさを横断させる**。これが一番効くと思う
2. **10%ごとのマイルストーン**。到達で指のビジュアルが段階変化する（ほじり力 → 爪が伸びて光る、パワー → 指が太くなる、速度 → 残像が出る）+ 称号 + ファンファーレ。0.5%は見えないが、10%は見える
3. **その場デモ**。購入直後、ショップの指がそのステータス固有のミニ演出をする
4. **想定飛距離の常時表示**。「小 33m / 中 42m / 大 51m / 特大 **--** / 伝説 **--**」をショップに出し、パワー不足のサイズは赤字か伏せ字にする。強化すると数字が動き、**やがて赤字が解ける**。§5.2 の「強化によって初めて活きる」を、プレイヤーに見える形で返せる

### (3) 飛ばしシーンの「世界」

**風・的・障害物は入れない。** 理由は、どれもRNGか正解タイミングを持ち込むので、企画が明確に否定している「タイミングを計るものではない、思い切り溜める快感」と衝突するため。当てにいく対象を置いた瞬間、ゲージは「計るもの」に戻ってしまう。

代わりに距離の見せ方に全部寄せる:

1. **背景の段階変化**: 机の上 → 部屋 → 窓の外 → 街 → 雲 → 成層圏 → 宇宙。段階到達で小さくテロップ
2. **距離マーカー**: 10m毎に線、100m毎に看板
3. **自己ベストの旗**: ベスト地点に旗を立て、超えた瞬間にヒットストップ + ファンファーレ + 旗をなぎ倒す
4. **サイズ別の自己ベスト**（記録が5本になる）。1本だけだと更新機会がすぐ枯れるが、5本あれば「伝説の記録がまだ0m」が次の動機になる

**通過オブジェクト**は笑い要員としてのみ置く。鳥・人・飛行機・衛星に当たると相手がリアクションして鼻くそが貼り付くが、**飛距離には一切影響しない**。減速させると罰になり、溜めた快感を台無しにする。

**下シーンは「やめる」ではなく「セッションリザルト」にする。** Webにプロセス終了はないので、ここを『今回ほじった数 / 最高飛距離 / 新規図鑑』を出す締めの画面にする。企画の「気が済むまでほじって飛ばして辞める、セッション型の遊び」に、終わりの一拍を与えられる。

## 10. 公開手順（GitHub Pages）

1. `git init` → GitHub に public リポジトリを作成
2. push
3. Settings → Pages → Source: `Deploy from a branch` → `main` / `(root)`
4. `https://<user>.github.io/<repo>/` で公開

**踏みやすい罠が2つある:**

- **パスは全部相対パスで書く**。サブディレクトリ配信になるので、`/src/main.js` は404になる。`./src/main.js` と書く
- **ES Modules は file:// で動かない**（CORSで弾かれる）。ローカル確認は `npx serve` か VS Code の Live Server を使う。Pages は http 配信なので本番は問題ない

## 13. 実装後に判明した、計画の誤り

計画書どおりに作って、動かしてみて壊れていた点。記録として残す。

| 計画の誤り | 何が起きたか | 直し方 |
|---|---|---|
| `reqPower[伝説] = 150`、`basePower = 20` | 強化は +0.5% の加算なので `playerPower` は現実的にせいぜい3倍（=60）。伝説の充足率は永久に0.27止まりで、**ショップの「やがて赤字が解ける」が嘘になる** | `reqPower` の幅を `basePower` の3倍以内（20〜60）に収めた |
| `upGrowth = 1.045` | 400回目の購入が10億コインになり、伝説の解禁が到達不能 | 1.012 に下げた |
| `angleCycle = 1.6s` | 正弦のメトロノームは45°が最速点。40〜45°の帯の通過が28ms（1.7フレーム）で運ゲー | 2.2s に下げた |
| エッジ遷移に再武装の概念が無い | ショップ→鼻に戻った指がエッジに残り、着いた瞬間に図鑑まで飛ぶ。ロック明けにも同じことが起きる | `armed` フラグを追加（§3.3） |

いずれも**紙の上では気づけず、実際に動かして初めて出た**。とくに上2つは、企画の「強化によって初めて活きる」という体験の核を、数値設定だけで静かに殺していた。
